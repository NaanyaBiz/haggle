# Contributing to haggle

Thank you for considering a contribution. This is a single-maintainer
hobby integration; the bar for contributions is high quality, narrow
scope, and a clear "why".

## Before you open a PR

1. **Open an issue first** for anything that isn't a one-line fix or a
   pre-existing issue you're claiming. The issue is the place to confirm
   the change is wanted before you spend time on a PR.
2. **Read `AGENTS.md`** — the canonical project documentation. The
   "AGL API — Key Facts" and "What NOT to Do" sections capture
   accumulated lessons from live debugging that are not obvious from the
   code alone.
3. **Run the dev loop locally** before pushing. CI runs the same checks
   and rejects PRs that fail any of them.

## Dev loop

```bash
uv sync                                                   # one-time setup
uv run pytest                                             # tests
uv run ruff check --fix custom_components/ tests/         # lint
uv run ruff format custom_components/ tests/              # format
uv run mypy custom_components/haggle                      # types (strict)
uv run pre-commit run --all-files                         # everything
```

`AGENTS.md → Dev Loop` has the full reference, including the local
hassfest invocation.

## Branches and worktrees

- `main` is always shippable. Never push to it directly — the
  `guard-main-branch` pre-commit hook will block you.
- Feature work happens in sibling worktrees at
  `/Users/dave/projects/haggle.wt/<branch>/` (or wherever your local
  setup lives). The `./scripts/wt new <branch>` helper sets one up.
- When you open a PR, target `main`. If a PR depends on another open PR,
  say so in the description.

## Commit conventions

Conventional Commits is enforced by the `commitlint` pre-commit hook:

```
feat: add daily consumption sensor
fix: handle missing consumption.values.quantity
chore(release): v0.2.0
ci: add hacs workflow
```

Every commit MUST also include a `Co-Authored-By: Claude` trailer. The
`require-claude-coauthor` hook enforces this. Most of this codebase was
generated by AI and reviewed by the human maintainer; the trailer is the
honest provenance.

```bash
git commit -m "feat: implement token rotation persistence

Co-Authored-By: Claude <noreply@anthropic.com>"
```

## PR checklist

Every PR that touches code (not pure CI/tooling fixes) must update:

- `CHANGELOG.md` — add bullet(s) under `## [Unreleased]` for every
  user-visible capability added, changed, fixed, or removed.
- `AGENTS.md` — Repo Map if files changed; AGL API section if endpoint
  facts changed; "What NOT to Do" if a footgun was discovered.
- `Closes #N` in the PR body if it resolves an open issue.

The `/pr` slash command (Claude Code) walks through this checklist and
opens the PR with a sensible body.

## Adding a new AGL endpoint

The "Contributing — Adding a New Endpoint" section in `AGENTS.md` has
the full procedure. Summary:

1. Identify the endpoint contract from your own AGL account using any
   HTTP-debugging tool.
2. Anonymise before committing — redact account number, contract number,
   address, and meter-read timeseries. Use the placeholders from
   `tests/fixtures/overview_response.json`.
3. Add an anonymised fixture, parser, client method, and tests.

Do **not** commit captures with real customer values.

## Security issues

See `SECURITY.md`. Please **do not** open public issues for suspected
vulnerabilities.
